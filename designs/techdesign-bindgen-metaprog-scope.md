# Tech Design: DOM `@extern` Bindgen via Body-Generating Rules

**Status:** design for review — no implementation yet. Closes
`designs/requests/accepted/request-bindgen-metaprog-scope.md`; grounded on
`docs/research-bindgen-metaprog-scope.md` (2026-07-19). Companion to
`designs/complete/techdesign-metaprog-phase4.md` (Layer D, landed) and
`designs/wasm-frontend/techdesign-06-bindgen-and-ship.md` §1 (the consumer).

**Grounding commit:** `5a80d0b`. Every mechanical claim below that is not marked
"[new]" was **verified live against `build/leviathan` at this tree** — the spike
transcripts are inlined at §2 and §4.3, not paraphrased. The one claim the research
left to "a spike, not more reading" (its Open Question 1) was run and **came back the
opposite of the research's headline** — see §2. That result is what picks the direction.

**Decision, up front:** adopt **path (c′)** — the research's path (c) (Layer-D body
rewrite of hand-declared stubs) corrected for the M32 obstacle it missed. Path (c′)
needs exactly **one** small, general new metaprog primitive — a body-*generating*
rewrite (`generates body of m`) that deliberately discards the original body — plus a
hand-written family of overloaded marshal helpers whose **overload resolution absorbs
the entire per-parameter slot-mapping problem** (gap 2's hard half) with zero
conditional templating. Paths (a) and (b) are rejected in §7 with reasons.

## 1. The ask, and the shape of the answer

techdesign-06 §1 wants the DOM handle-wrapper marshaling bodies — today hand-written in
`kPreludeWasm` (`Resolver.cpp:5507-5592`) — generated by the rules engine, so that the
op-string and native-tag data live as declared data rather than as fourteen
copy-pasted `std::sysHost{I,S,V}(op, h0, h1, a, b)` call bodies. The research
(`docs/research-bindgen-metaprog-scope.md`) decomposed the blockage into two
independent gaps:

- **Gap 1 — declare-vs-generate collision.** A rule that matches a class and injects a
  same-named member back into it trips the M33 confluence check
  (`Rules.cpp:2515-2550`). You cannot *add* the generated `setAttr` beside the
  hand-declared `setAttr`.
- **Gap 2 — the target moved + per-method branching.** The marshaling body varies per
  method along axes `$for` templating cannot express in one call: native tag (I/S/V),
  the op string, receiver handle (`this.h` vs `0`), and which parameter fills which of
  the fixed slots `h1`/`a`/`b`.

The answer this design commits to:

| Problem | Resolution in path (c′) |
|---|---|
| Gap 1 (placement) | **Layer D rewrites the *body* of the existing declaration in place** — no new `Stmt` is ever added to `cls->body`, so M33 is never consulted. The signature stays hand-declared; only the body is generated. |
| Gap 2 — native tag + op string + receiver | An explicit **`@extern(op:…, kind:…)`** attribute carries the two facts that are not derivable; one rule per `kind` (a `where` filter) selects the body template. Landed machinery. |
| Gap 2 — **argument-slot mapping** (the genuinely hard half) | **Overload resolution over a small hand-written marshal-helper family.** The rule forwards the method's own parameters verbatim with `$_args`; the compiler's existing overload resolution picks the helper overload whose signature matches the parameter shape, and *that overload* places each argument in the correct `h1`/`a`/`b` slot. Zero conditional templating in the rule. |
| The new capability gap the research missed | **M32 forbids a `replace` that drops the original body.** Path (c) as sketched does not compile. Path (c′) adds one general primitive — `generates body of m` — that makes deliberate whole-body generation legal (§4). |

Net authoring change to the DOM surface: the ~9 distinct hand-written marshal call
bodies collapse to a **hand-written helper family of ~9 overloads written once**, plus a
**one-line `@extern(...)` attribute per generated method** whose body is then machine
-produced. This satisfies the ticket's path-(b)/(c) acceptance ("doc 05's hand-written
stub bodies are deleted in favor of generated ones; signatures may remain
hand-declared").

## 2. The load-bearing spike — why "zero new primitives" is wrong (M32)

The research's headline (its §7.3/§7.4) is that path (c) closes both gaps **with zero
new engine primitives**, and its Open Question 1 flags the one unverified assumption:
*does a method whose body is only ever filled by a `rewrites` rule compile?* It answers
from mechanism and asks for a spike. **The spike was run. It fails.**

Layer D's `replace` template is governed by **M32** (`Rules.cpp:735-748`): `$body` must
appear in the template **exactly once**. Zero occurrences is a hard error, by design —
the engine treats a body-dropping `replace` as "silent obliteration" and refuses it:

```
$ build/leviathan --run m32_spike.ext
m32_spike.ext:3:5: error: rule 'genBody' never references '$body' — a 'replace'
that drops the original body is silent obliteration; splice '$body' to keep it
      rule genBody rewrites body of m {
      ^~~~
1 error(s)
```

(spike source: a `rewrites body of m` whose `replace` synthesizes a fresh
`std::sysHost…` call and never mentions `$body`.)

This is fatal to path (c) *exactly as the research sketched it*, because a faithful
bindgen's whole point is to **replace** the hand-written marshaling body, not wrap it —
there is no meaningful "original body" to splice back. M32's intent is right (don't let a
careless rule silently eat a method), but bindgen is the legitimate case M32 has no
exemption for. Hence path (c′) needs the one primitive in §4: an *explicit* body-drop
that turns M32's "silent obliteration" into sanctioned, opt-in obliteration.

Everything else the research claimed to be landed **was verified to actually be landed**
(§4.3 transcripts): overload-resolution slot mapping, `$_args` forwarding inside a
`rewrites`/`generates` template, attribute-value splicing (`$e.op` as a bare
expression — *not* `"$e.op"` inside a string literal, which does not interpolate), and
`where`-clause dispatch across multiple body-rewrite rules. So the design surface reduces
to precisely one new thing.

## 3. Architecture, end to end

### 3.1 The marshal-helper family (hand-written once)

Add, next to the three native seams in `namespace std` (unchanged) and inside
`namespace Dom`, a family of overloaded helpers. Each is a one-line `=>` method funnelling
into `sysHost{I,S,V}` with the slots already placed. This is the *only* hand-written
marshaling logic that survives; it is written once, not per DOM method.

```leviathan
namespace Dom {
    // --- receiver-less constructors (document-scoped ops, h0 = 0) ---
    DomNode __new(string op)                       => DomNode(std::sysHostI(op, 0, 0, "", ""));
    DomNode __new(string op, string a)             => DomNode(std::sysHostI(op, 0, 0, a,  ""));
    int     __int(string op)                       => std::sysHostI(op, 0, 0, "", "");

    // --- receiver-handle queries (h0 = this.h), string return ---
    string  __str(int h, string op)                => std::sysHostS(op, h, 0, "", "");
    string  __str(int h, string op, string a)      => std::sysHostS(op, h, 0, a,  "");

    // --- receiver-handle actions (h0 = this.h), void return ---
    void    __act(int h, string op)                => std::sysHostV(op, h, 0, "", "");
    void    __act(int h, string op, string a)      => std::sysHostV(op, h, 0, a,  "");
    void    __act(int h, string op, string a, string b) => std::sysHostV(op, h, 0, a, b);
    void    __act(int h, string op, DomNode child) => std::sysHostV(op, h, child.h, "", "");
}
```

The four `__act` overloads are the crux: **their signatures are the slot-mapping table.**
`(int, string)` → zero args → `a=b=""`; `(int, string, string)` → first string → `a`;
`(int, string, string, string)` → two strings → `a,b`; `(int, string, DomNode)` → the
handle param → `h1=child.h`. Adding a hypothetical two-handle DOM method later that has
no matching overload produces a *compile error at generation time*, not a silent wrong
marshal — this is the enforcement the research worried a "positional convention" would
lack (research §7.4 closing paragraph).

### 3.2 The per-method `@extern` annotation

```leviathan
namespace Dom {
    enum MarshalKind { Node, Int, Str, Fluent, Act }   // [new] if enum-typed attr args
    attribute extern { string op; MarshalKind kind; }  //       land; else `string kind`
}
```

`op` is the host op string (`"setAttribute"`), not derivable from the method name (§5).
`kind` is the one bit that is *not* recoverable from the signature — specifically it
disambiguates the two `DomNode`-returning shapes that share a return type: `Node`
(construct a fresh wrapper from the native result) vs `Fluent` (ignore the native result,
`return this`). The five kinds and their exhaustive mapping to the current surface:

| `kind` | native | receiver | returns | DOM methods |
|---|---|---|---|---|
| `Node` | I | `0` | new `DomNode` | `Dom::body`, `create`, `textNode`, `byId` |
| `Int` | I | `0` | `int` | `Dom::listenerCount` |
| `Str` | S | `this.h` | `string` | `DomNode.attr`, `text`; `DomEvent.type`, `targetValue` |
| `Fluent` | V | `this.h` | `this` (`DomNode`) | `DomNode.setAttr`, `setText` |
| `Act` | V | `this.h` | `void` | `DomNode.append`, `dispatch`, `release` |

Every marshaled method in `kPreludeWasm` maps to exactly one kind; the four excluded
methods (`exists`, `click` — pure Leviathan; `on`/`off`/`events` — closures, out of v1
scope per techdesign-06 §1) carry no `@extern` and are untouched by the rules.

### 3.3 The rule set — one `generates` rule per kind

```leviathan
namespace Dom {
    rule bindNode   generates body of m { match @extern(e) on method m where e.kind == Node
        replace `return Dom::__new($e.op, $_args);` }

    rule bindInt    generates body of m { match @extern(e) on method m where e.kind == Int
        replace `return Dom::__int($e.op, $_args);` }

    rule bindStr    generates body of m { match @extern(e) on method m where e.kind == Str
        replace `return Dom::__str(this.h, $e.op, $_args);` }

    rule bindFluent generates body of m { match @extern(e) on method m where e.kind == Fluent
        replace `Dom::__act(this.h, $e.op, $_args); return this;` }

    rule bindAct    generates body of m { match @extern(e) on method m where e.kind == Act
        replace `Dom::__act(this.h, $e.op, $_args);` }
}
```

Five rules, each branch-free: no `$for`, no `$if`, no per-param logic. `$e.op` splices the
op string; `$_args` forwards the matched method's own parameter list verbatim into the
helper call, where overload resolution finishes the job. `where e.kind == Node` partitions
the annotated methods across the five rules (dispatch confirmed live, §4.3). `Node`/`Int`
rules name no `this.h` (they fire on `Dom` namespace functions, which have no receiver);
`Str`/`Fluent`/`Act` reference `this.h` (they fire on `DomNode`/`DomEvent` methods, which
declare `int h`). `method` and `function` are the same node shape to the subject matcher
(`Rules.cpp:1317-1326`), so the same `on method m` clause matches both.

### 3.4 Worked expansions

The DOM surface is authored as signature + one-line `@extern` + a **type-valid placeholder
body** (the body the rule discards; see §4 on why it must parse):

```leviathan
class DomNode {
    int h;
    new DomNode(int handle) { h = handle; }
    bool exists() => h != 0;                                    // untouched (no @extern)

    @extern("setAttribute", Fluent) DomNode setAttr(string name, string value) => this;
    @extern("getAttribute", Str)    string  attr(string name)                  => "";
    @extern("appendChild",  Act)    DomNode append(DomNode child)              => this;
    @extern("getText",      Str)    string  text()                            => "";
    @extern("dispatchEvent", Act)   void    dispatch(string type) { }
    @extern("release",      Act)    void    release() { }
    // ... setText (Fluent), events/on/off/click/exists stay hand-written ...
}
namespace Dom {
    @extern("createElement", Node) DomNode create(string t) => DomNode(0);
    @extern("cloCount",      Int)  int listenerCount()      => 0;
    // ... body, textNode, byId ...
}
```

expands (via `generates` + overload resolution) to bodies byte-equivalent in behavior to
the hand-written ones:

| method | `$_args` | generated body | resolves to | native call (matches §3 table) |
|---|---|---|---|---|
| `setAttr(name,value)` | `name, value` | `Dom::__act(this.h,"setAttribute",name,value); return this;` | `__act(int,string,string,string)` | `sysHostV("setAttribute", h, 0, name, value)` |
| `attr(name)` | `name` | `return Dom::__str(this.h,"getAttribute",name);` | `__str(int,string,string)` | `sysHostS("getAttribute", h, 0, name, "")` |
| `append(child)` | `child` | `Dom::__act(this.h,"appendChild",child);` | `__act(int,string,DomNode)` | `sysHostV("appendChild", h, child.h, "", "")` |
| `text()` | *(none)* | `return Dom::__str(this.h,"getText");` | `__str(int,string)` | `sysHostS("getText", h, 0, "", "")` |
| `create(t)` | `t` | `return Dom::__new("createElement",t);` | `__new(string,string)` | `DomNode(sysHostI("createElement",0,0,t,""))` |
| `listenerCount()` | *(none)* | `return Dom::__int("cloCount");` | `__int(string)` | `sysHostI("cloCount",0,0,"","")` |

Note `append`: the handle parameter lands in `h1` **purely because overload resolution
selected the `DomNode`-param overload** — the rule did nothing special. That is gap 2's
"which param goes in which slot" branch, resolved by the type system instead of by
template control flow.

### 3.5 Acceptance is behavioral, not textual

The generated body is `Dom::__act(this.h, "appendChild", child)`, not the literal
`std::sysHostV("appendChild", h, child.h, "", "")`. They are behaviorally identical (the
`=>` helper is a single-expression forwarder the backend inlines), so `--run` and the wasm
DOM pins are byte-identical; `--ir` differs only by an inlinable call the optimizer folds.
The acceptance criterion "diff-match the hand stubs" is therefore realized as *behavioral*
equivalence (§12), with `--expand` showing the generated source for human review — not a
requirement that the emitted call text match character-for-character (which the seam's own
funnel-through-a-helper shape never guaranteed anyway).

## 4. The one new primitive — `generates body of`

### 4.1 What it is

A body-rewrite anchor that **replaces** a callable member's body wholesale and **discards**
the original — the sanctioned form of what M32 forbids by default. It is a sibling of the
landed `rewrites body of`, differing in exactly two rules:

- **`$body` is unavailable.** There is no original to splice; referencing `$body` in a
  `generates` template is a new error **M36** (§10). This preserves M32's spirit: dropping
  the original is never *silent*, it is spelled `generates` at the rule head.
- **The `replace` template need not (and by M36 must not) reference `$body`.** M32's
  "exactly once" check is skipped for `generates` rules and replaced by M36's "never."

Everything else is Layer D verbatim: target resolved via the match's own bindings
(`boundClass`/subject), member must be callable-with-a-body, body overwritten in place at
`subj->memberBody` (`Rules.cpp:1811-1854`), no new `Stmt` in `cls->body` (so **M33 never
fires** — gap 1 gone), runs in Layer D's pass-2 rewriter phase after additive injects.

### 4.2 Grammar and implementation delta

Parser (`Parser.cpp`, the `rewrites` production): accept `generates` as a second
body-rewrite keyword, setting `AnchorKind::BodyGenerate` (a new enum sibling of
`BodyReplace` in `Ast.hpp:293-302`). Grammar is otherwise identical:

```
rule NAME generates body of <bind> {
    match ...
    replace `...`            // no $body; M36 if present
}
```

`Rules.cpp` changes, all small:
- `checkRewriteRules` (the M32 site, `Rules.cpp:735-748`): branch on the anchor —
  `BodyReplace` keeps the "exactly once" check; `BodyGenerate` asserts "zero" and raises
  **M36** on any `$body` reference.
- `expand()` (`Rules.cpp:1811`): `BodyGenerate` takes the same overwrite path as
  `BodyReplace` but leaves `replaceBodyOrig_` null (so any stray `$body` splice attempt is
  already the M36-guarded case, never reached at expand time).
- Confluence: two `generates`/`rewrites` rules targeting one body is the existing M35
  double-rewrite conflict (`Rules.cpp:72` neighborhood) — no new confluence logic; a
  method may be body-written once. The five DOM rules are mutually exclusive by
  `where e.kind`, so no method is claimed twice.

This is a **general** capability, not DOM-specific: any future bindgen (FFI, a serializer,
a proxy generator) that needs "declare the signature, machine-fill the body" uses it. It is
strictly smaller than either rejected path's primitive (§7).

### 4.3 Grounding transcripts (run at `5a80d0b`)

The four landed mechanics `generates` composes with, each verified against
`build/leviathan`:

**(a) Overload resolution performs the slot mapping** — the generated-body shapes from
§3.4, hand-written and run:
```
setAttribute(7,0,'id','x')      # __act(h,op,string,string) → a,b
getAttribute(7,0,'id','')       # __str(h,op,string)        → a
getText(7,0,'','')              # __str(h,op)               → none
appendChild(7,9,'','')          # __act(h,op,DomNode)       → h1=child.h
```

**(b) `$_args` forwards the matched method's params inside a body-rewrite template**, and
**(c) attribute values splice as bare expressions** (`$o.name`, not `"$o.name"`):
```
$ build/leviathan --run rewrite_args2.ext     # replace `return M::sink($o.name, $_args) + $body;`
SETATTR|x|y<orig>
```

**(d) `where` on an attribute arg dispatches multiple body-rewrite rules** (the §3.3
kind partition):
```
$ build/leviathan --run kind_dispatch.ext     # asS where o.kind=="S"; asV where o.kind=="V"
S:getAttribute
V:setAttribute
```

**(e) A rule fires on annotated classes declared in its own namespace/file, no `uses`**
(so the Dom surface + rules co-locate, §6):
```
$ build/leviathan --run samefile.ext
op=getAttribute
```

Only `generates` itself (§4.1-4.2) is unbuilt; every mechanic it leans on is demonstrated.

### 4.4 Why a placeholder body, not a bodyless declaration

`generates` overwrites an *existing* body; the method must therefore carry one at parse /
pass-1, before the rule stage runs. The §3.4 placeholders (`=> this`, `=> ""`,
`=> DomNode(0)`, `{ }`) are type-valid throwaways. This deliberately avoids a
*language*-level change (permitting bodyless concrete methods), which has a far larger
blast radius than a metaprog-local anchor. Two mitigations keep the placeholder cost
negligible:
- Where the Dom surface lives in the non-checked prelude segment
  (`Resolver.cpp:5597` "the checker never walks the prelude"), placeholder validity is
  moot anyway.
- Optional follow-on (deferred, not in this ticket): allow `generates` to target a
  member declared with a `;`-terminated signature-only body *when* it carries the
  triggering attribute — a small parser affordance that removes placeholders entirely.
  Filed as a nicety, not a blocker.

## 5. How this closes gap 1 and gap 2

**Gap 1 (declare-vs-generate collision) — dissolved, not solved.** `generates body of m`
never adds a member; it overwrites `m`'s body in place. `injectMember`'s M33 loop
(`Rules.cpp:2515-2550`) iterates only over *newly injected* members and is never entered.
No interface split, no cross-class injection, no `meta::Class.bases`-as-objects — the
entire path-(a) apparatus the request ticket contemplated is unnecessary. Signature and
generation-data live on the same declaration in the same class, which is exactly what
Layer D (unlike `member of` injection) permits.

**Gap 2 — split into its three sub-parts, each closed by landed machinery:**

1. *Op string* (`setAttr`→`"setAttribute"`, `release`→`"release"` — no uniform derivation,
   research §5 item 1): carried explicitly as `@extern(op:…)`. One string per method.
2. *Native tag* (I/S/V) + the `Node`/`Fluent` return-shape ambiguity: carried as
   `@extern(kind:…)`, a 5-value enum, dispatched by five `where`-filtered rules.
3. *Argument-slot mapping* (the research's "gap 2's real content", §5 item 3 — the
   `$for`-can't-branch problem): **not expressed in the template at all.** `$_args`
   forwards the raw parameter list; overload resolution over the §3.1 helper family
   assigns each argument its `h1`/`a`/`b` slot. The "positional convention" the research
   described (§3: string params fill a then b; a handle param fills h1) is realized *as a
   set of overload signatures*, and is therefore **type-checked and enforced** — a future
   method that breaks the pattern gets a no-matching-overload compile error, not a silent
   miscompile. This is strictly better than both options the research left open for OQ2
   (implicit convention = unenforced; explicit per-param slot attributes = verbose and
   string-typo-prone).

## 6. Placement — prelude string vs shipped `.lev` file

`kPreludeWasm` is a C++ string literal that the checker does not walk
(`Resolver.cpp:5597`). Whether the rule engine runs over it is **the one placement
question a spike must settle** (§13) — rules demonstrably run over ordinary checked files
(§4.3(e)) but the prelude's not-checked status makes its rule participation unproven.

Two placements, and a recommendation that dovetails with an already-accepted ruling:

- **(Recommended) Move the Dom surface into a shipped `dom.lev` stdlib file.** The
  `stdlib-ship-as-files` ruling (accepted;
  `designs/requests/accepted/request-stdlib-ship-as-files.md`, and the memory note of the
  same ruling) already commits the stdlib to `.lev` files rather than in-binary segments.
  The Dom surface is browser-only and per-target — a natural first citizen of that scheme.
  As a normal checked file, rules over it are the demonstrated §4.3(e) case, and
  placeholder bodies are ordinary type-checked code. This also unblocks techdesign-06 §2's
  per-target consumption story (that doc's §2 STOP was the same ship-as-files ruling; it is
  now resolved).
- **(Fallback) Keep it in `kPreludeWasm`** and confirm-or-add rule-stage processing of the
  prelude segment. Viable if the ship-as-files migration of *this* surface slips, but it
  couples the bindgen to prelude internals and needs the §13 spike to pass first.

Either way the `@extern` rules and the annotated Dom classes live in **one** file/segment
(§4.3(e) proves co-location needs no `uses`), so there is no cross-file rule-visibility
question.

## 7. Alternatives considered and rejected

### 7.1 Path (a) — interface/inherited method reflection + cross-class injection

The request ticket's first option. Needs (research §7.1): `meta::Class.bases` as resolved
objects (item H), `C.methods` unioning inherited/interface signatures (item E extended to
methods), **and** a brand-new cross-class injection primitive with no partial
implementation. Three sizeable surface extensions — and it *still* leaves gap 2 entirely
unsolved (interface reflection tells you a method exists, not its op-string/tag/slots).
Rejected: largest lift, worst risk, does not even close gap 2. Only revisit if a future
requirement is *fully* interface-generated methods with zero hand-declared signatures — a
stronger "faithful bindgen" than this ticket asks for.

### 7.2 Path (b) — attribute-driven generation via `member of` injection

The request ticket's second option, taken literally (annotate methods, `inject … at member
of C`). Rejected because it walks straight into gap 1: injecting the generated
`setAttr` beside the hand-declared `setAttr` is the M33 collision (research §4, verified
live). Path (b)'s *data model* (op/kind on the attribute) is correct and is exactly what
(c′) adopts — but its *placement mechanism* is wrong. The request ticket's phrasing that
(b) "sidesteps interface reflection or cross-class injection entirely" is true only once
(b) is paired with a Layer-D placement, i.e. once it becomes (c′).

### 7.3 Path (c) — Layer-D rewrite that drops the body, zero new primitives

The research's recommendation. **Rejected as non-compiling** — M32 forbids the
body-dropping `replace` it depends on (§2, spiked). (c′) is (c) plus the minimum primitive
(`generates`) that makes it legal. All of (c)'s architecture survives; only the "zero new
primitives" claim does not.

### 7.4 Zero-primitive escape hatches (why none is preferable to `generates`)

- *Relax M33 instead of M32* (let an injected member replace a hand-declared one): touches
  confluence/collision semantics — riskier and broader than a body-local M32/M36 tweak.
- *`distinct` twins* (keep hand-written and generated side by side): yields two methods,
  not one; useful only as a transitional diff-check (§12), never as the end state.
- *Keep hand-writing the marshal call as the `$body` and have the rule wrap it*: defeats
  the entire purpose (the marshal call is what we want to stop hand-writing).

`generates` is the smallest, most contained, and most reusable of every option on the
table.

## 8. Generality and future surface

- **`generates body of` is a general bindgen primitive.** Nothing about it is DOM-specific.
  Any "author the signature, machine-fill the body" generator — a C FFI shim, a
  `Serializable` codec, a logging proxy, a null-object stub — is now expressible. This is
  the reusable capability the ticket's functionality is "expected to" yield beyond the
  immediate DOM need.
- **The overload-family pattern generalizes too.** Encoding a marshaling ABI's slot
  discipline as a set of overload signatures — and letting the type checker enforce it — is
  a reusable technique for any fixed-arity reflective seam, not a DOM one-off.
- **Graceful failure on surface growth.** A new DOM method whose parameter shape has no
  matching `__new/__str/__act` overload fails at generation with a no-overload error naming
  the method — the maintainer adds one overload (one line) or hand-writes that method. No
  silent breakage; the §3.1 family is the single, visible extension point.
- **Collapsing the five rules to one (deferred).** If `request-metaprog-splices.md` item
  (B) (identifier-position splicing of comptime strings) ever lands, `Node`/`Int`/`Str`
  and the `__new/__int/__str/__act` selection could be driven by a single templated rule.
  This design deliberately does **not** request (B) — five near-identical rules are cheaper
  than a new splice primitive (research OQ3), and each rule reads clearly. Noted as a
  future consolidation, not a dependency.

## 9. Out of scope

- Closure-bearing methods `on`/`off`/`events` (the event-stream seam) — hand-written,
  per techdesign-06 §1's own v1 scope discipline.
- Pure-Leviathan conveniences `exists()`, `click()` — no `@extern`, untouched.
- The full P4 deferred tail (`techdesign-metaprog-phase4.md` §9.1-9.8) — this design needs
  none of it; only the `generates` anchor (§4) is new.
- Structs-by-value across the bridge, non-string/non-handle parameter types — no consumer
  yet; overload-family error (§8) guards the boundary.
- Any change to the three native seams (`sysHost{I,S,V}`) — a frozen HARD boundary
  (hard-06); this design sits entirely above it.

## 10. Diagnostics catalog

| Code | Trigger | Message (shape) |
|---|---|---|
| **M36** [new] | `$body` referenced in a `generates` template | `rule '…' is a 'generates' rule — the original body is discarded, so '$body' is unavailable; use 'rewrites' to keep it` |
| M32 (existing) | `$body` count ≠ 1 in a `rewrites` template | unchanged; now scoped to `rewrites` only |
| M33 (existing) | injected member collides with an existing one | unchanged; **never reached** by this design (no member is injected) |
| M35 (existing) | two body-rewrites target one body | unchanged; the five DOM rules are `where`-exclusive so it never fires here |
| (existing) | `generates`/`rewrites body of` names a non-callable or bodyless target | reuse Layer D's existing "target must be callable-with-a-body" diagnostic |

The only new diagnostic is M36 — the guard that keeps body-dropping explicit rather than
silent (the property M32 was protecting).
