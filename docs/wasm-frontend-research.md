# WebAssembly front-end target (Leviathan in the browser, DOM access) — technical research dossier

**Status:** pre-design research. **Not** a tech design. This is the complete technical
substrate a tech design for the wasm-browser target
(`designs/requests/proposal-wasm-frontend.md`) must build on: exactly what exists in the tree
today, how each substrate the target rides is actually shaped, which of the proposal's stated
blockers have since been resolved by other landed work, which remain open, and what a real
implementation touches at the level of `file:line`. Every implementation claim below is
anchored to code verified against the working tree (`agent2`, audited 2026-07-16), not to
prose in `info.md` / `docs/reference.md` / the proposal — those are cited only for *intent*.

Audience: whoever writes the tech design (a Fable-class author), and the implementers after
them. This is intended to be the **only** document needed to write the wasm-target design.

The companion is the proposal itself (`designs/requests/proposal-wasm-frontend.md`, 2026-07-08),
which records the *verdict* ("it's libraries + a backend + a floor retarget, the language and
IR are untouched; defer the real work past Gate E"). This dossier does not relitigate that
verdict — it grounds it, and it reports that **the ground has moved**: three of the four
"moving substrates" the proposal named as reasons to wait (§4 there) have landed in the eight
days since it was written. See §12.

---

## 0. One-paragraph orientation

The wasm-browser target is **three artifacts and zero language changes**: (1) a new backend
column, IR → `wasm32`, which the LLVM backend already reaches for free — `emitObject` takes a
target triple, initializes *all* LLVM targets for any cross-triple, and links against an
LLVM 18 built with the `WebAssembly` target (§4); (2) a browser implementation of the Layer-0
platform floor — one new file `lv_plat_wasm.c` implementing the same ~40 symbols
`runtime/lv_plat.h` declares, with syscalls swapped for host imports (§6); (3) a JS/DOM
bridge — a single reflective marshaler over the frozen `LvValue {tag, payload}` ABI plus a
closure trampoline, generated by the rules engine (§8). The central claim — *everything
lowers to one IR (`src/Ir.hpp`, 44 opcodes); a backend only diverges on how it emits each op;
the wasm target is another column in the engine-coverage matrix* — is grounded and correct
(§2). The two runtime-coupled hard parts the proposal flagged as the reason to defer were
**async suspension** and **per-target stdlib packaging**. As of this audit, **async
suspension has fully landed** — true stackful tasks on all three engines, the uncolored
`await` surface unchanged (§7), which is precisely the model JSPI realizes engine-side in a
browser — leaving **per-target stdlib packaging as the single genuinely-unsettled upstream
dependency** (§12). The spike the proposal asks for (LLVM→`wasm32`, run pure computation,
differential-match the oracle) is now a nearly-mechanical afternoon, because the runtime ABI
and archive it needs also landed (§5).

## 1. Terminology, pinned

| term | meaning here |
|---|---|
| **the floor / Layer 0** | the thin platform-primitive interface every backend's runtime bottoms out on. Two planes: the C **platform floor** (`runtime/lv_plat.h`, ~40 functions, one impl per OS) and the language-exposed **floor natives** (`sys*` names). A wasm target replaces the former. |
| **the ABI** | the frozen calling/representation contract in `runtime/lv_abi.h`: every value is a 16-byte `LvValue {int64_t tag; int64_t payload}`, crossing the generated↔runtime boundary *only* as `LvValue*`. This is what the wasm↔JS marshaler is written against, once. |
| **tagged value** | the `[tag(8)][payload(8)]` pair. Tag numbering is the `LV_*` set (0–12, `lv_abi.h:38-53`), **not** the interpreter's `VKind` order (`RuntimeValue.hpp:36`) — they differ and conflating them "corrupts every array/closure dispatch" (`lv_abi.h:55-58`). |
| **shape** | a class's slot table. Runtime descriptor `LvClassInfo` (`lv_abi.h:227-246`): `slotNames[i]` is the field at slot `i`; object body places slot `i` at byte offset `16 + 16*i`. The reflective marshaler walks this. |
| **the marshaler** | the single table-driven routine that moves an `LvValue` across the wasm↔JS boundary by reading its tag and (for objects) its shape. Written *once* because the representation is uniform — the whole §2 argument. |
| **the bridge / bindgen** | the `wasm-bindgen` equivalent: a handle table for opaque JS values, a closure trampoline for DOM events, and rules-engine-generated import declarations + marshaling stubs. A library + rules artifact, not a compiler subsystem (§8). |
| **capability subset** | *not* a language subset. A per-target statement of which platform capabilities are in scope, expressed through the existing engine-coverage matrix (§2). Absent capabilities (files, processes in a browser) are compile-time diagnostics; present-but-different ones (console, time) are a silent floor swap. |
| **JSPI** | JavaScript Promise Integration — a WebAssembly *engine* feature (V8 et al.) that suspends and resumes a whole wasm call stack when a suspending import returns a JS promise. The browser realization of the stackful suspension Leviathan already does natively (§7). |
| **the spike** | the proposal's §9 de-risking action: retarget LLVM→`wasm32`, run a pure-computation corpus subset, differential-match the oracle. Commits to nothing; validates the type-system claim empirically (§5). |

## 2. The central claim, grounded — one IR, the backend is a column

The proposal's load-bearing structural claim is *"everything lowers to one IR (info.md §17);
the IR is the semantic truth and each backend only diverges on 'emit for an op.'"* This is
literally true in the tree.

### 2.1 The IR is one small enum — the entire surface a wasm backend emits for

The whole instruction set is a single `enum class Op` in `src/Ir.hpp:26-89` — **44 opcodes**
across eight categories. The header's own framing (`src/Ir.hpp:11-19`) states the thesis:
*"The bytecode IR (§17): the single semantic truth both variants share… Method/accessor/
constructor BODIES — including the in-language stdlib from the prelude — are themselves
lowered to IR functions."*

```cpp
enum class Op {
    // data
    LoadConst, Default, Move, MoveClear, CopyVal, Arith, Not, Neg,
    // control
    Jump, JumpIfFalse, JumpIfTrue, Ret, RetVoid, Throw,
    // calls
    Call, CallDyn, CallValue, CallNativeFn,
    // objects
    NewObject, GetMember, SetMember, RawGet, RawSet, RawGetWeak, RawSetWeak,
    // collections
    NewArray, NewArraySized, NewMap, NewBlock, NewBlockStr, MakeRange,
    GetIndex, IndexStore, IterLen, IterAt, ColGet, IsType, Await,
    LoadGlobal, StoreGlobal,
    // closures
    MakeClosure, CaptureVar, LoadCapture,
    // output
    Print, PrintNl,
    // §15 escaping-tier ARC
    VFree,
};   // src/Ir.hpp:26-89 (categories are the source's own comments)
```

The AST→IR funnel every backend sits behind is `Lowerer::lower(Program&, Program& prelude,
IrModule&)` (`src/Lower.hpp:26`); constructs outside coverage produce a clean *"IR: not yet
lowerable: …"* diagnostic (`src/Lower.hpp:10-11`). The prelude's entire in-language stdlib is
lowered "like any other code" (`src/Lower.hpp:22-24`) — which is why a whole web-foundations
layer (JSON, DateTime, digests, HTTP hardening) shipped with **zero new natives and zero
compiler changes**, every engine getting it via lowering (`techdesign-09-web-foundations.md:12-14`).
That is the strongest existing proof of the "one IR" thesis: a wasm backend that correctly
emits these 44 ops inherits the entire language and stdlib for free.

Note the one backend-visible staged flag: `IrModule::columnar` (`src/Ir.hpp:130`) — the
struct-of-arrays layout gate. "Interpreters ignore it (they stay boxed)." A wasm backend can
too, initially (relevant to the zero-copy typed-array path, §8.4 and §13 risk #7).

### 2.2 The engine-coverage policy — a wasm target is another matrix column

The policy the proposal invokes is stated verbatim at `techdesign-00-overview.md:187-190`:

> Engine-coverage policy follows the existing pattern: **oracle + IR are mandatory** for
> everything; **emit-C++** for everything except the event-loop/system layer; **ELF** for the
> whole language (deferred items need an explicit diagnostic, never silence); **LLVM**
> scalar-core only (extend when trivial, else report uncovered).

The existing backends, each self-describing in its header:

| backend | file | role |
|---|---|---|
| **tree-walk oracle** | `src/Eval.cpp` (`Eval.hpp:12-16`) | semantics oracle over the AST; "not the shipped interpreter… just correct." |
| **IR interpreter** | `src/IrInterp.cpp` (`IrInterp.hpp:9-11`) | executes an `IrModule` directly; observable behavior identical to the oracle. The differential anchor. |
| **emit-C++** | `src/CGen.cpp` (`CGen.hpp:10-16`) | IR → freestanding C++; "Ops outside coverage fail with a diagnostic." |
| **pure ELF** | `src/X64Gen.cpp` (`X64Gen.hpp:11-14`) | IR → x86-64 machine code + ELF, no g++/linker/libc. **Frozen** (`techdesign-portable-backend.md:14-23`). |
| **LLVM** | `src/LlvmGen.cpp` (`LlvmGen.hpp:6-19`) | IR → LLVM IR → object file; values cross as `LvValue*`; calls the runtime's `lvrt_*` C functions. The wasm target's host. |

A wasm-browser backend is **another column**: it declares its covered capability subset,
implements the browser-relevant floor, and cleanly diagnoses the rest — exactly the
established ELF-DNS pattern (§6.4). This is the whole reason the language and IR are untouched.

### 2.3 Why the type-system objection is misplaced (the premise the proposal corrects)

The instinct that "Leviathan erases like Java, so it can't hand typed values to a
`wasm-bindgen`-style marshaler" is wrong on both facts and requirement, and the tree confirms
why:

- **Every value is self-describing by tag** — the 16-byte `LvValue` carries an explicit `tag`
  (`lv_abi.h:36`), so the marshaler recovers the runtime type of any value with zero static
  call-site knowledge.
- **Every object knows its shape** — an object payload leads with `classId`
  (`lv_abi.h:95-96`), and `LvClassInfo` (`lv_abi.h:227-246`) maps `classId` → `slotNames[]`,
  so the marshaler enumerates typed slots reflectively.

Rust needs per-instantiation `wasm-bindgen` glue precisely because it has **no runtime and no
RTTI** — the static call-site type is its only handle. Leviathan is the opposite: it marshals
*reflectively*, off tag + shape, so the marshaler is written **once**, table-driven, not
regenerated per instantiation. Monomorphization is a workaround for having no runtime, not a
prerequisite for a wasm↔JS boundary — every dynamic-runtime language ships on wasm with full
DOM access over a uniform representation (C#/Blazor, Kotlin/Wasm, Dart, Pyodide). The uniform
representation is a *feature* at the boundary: one marshaler, not N.

## 3. The value model & ABI — the marshaling substrate

The marshaler is built against the **compiled ABI**, `runtime/lv_abi.h` — *not* the
interpreter's fat `Value` union (`src/RuntimeValue.hpp:40-53`). These are two distinct models
with different tag numberings; the wasm bridge only ever sees the former.

### 3.1 The tagged 16-byte value and the boundary rule

```c
typedef struct LvValue { int64_t tag; int64_t payload; } LvValue;   /* lv_abi.h:36, 16 bytes */
```

The **boundary rule** (`lv_abi.h:6-10`) is the single most important fact for a wasm calling
convention:

> Boundary rule (§2.1, hurdle H-1): every value crosses the generated↔runtime boundary as an
> `LvValue*` pointer (out-param first, then inputs). No function here passes or returns
> LvValue by value — clang lowers 16-byte structs differently per target (SysV splits into
> two scalars, Win64 passes by pointer); the pointer convention sidesteps that entirely.

For wasm this is doubly convenient: wasm has no SysV/Win64 struct-splitting divergence, but
the pointer convention is already the contract, so the wasm marshaler passes a linear-memory
offset (`i32` under `wasm32`) to an `LvValue` and reads `tag`/`payload` from it. No ABI work.

The tag set (`lv_abi.h:38-53`):

```c
enum { LV_VOID=0, LV_INT=1, LV_FLOAT=2, LV_BOOL=3, LV_STR=4,
       LV_OBJ=5, LV_ARR=6, LV_MAP=7, LV_NONE=8, LV_CLO=9,
       LV_CHAR=10, LV_BLOCK=11, LV_WEAK_PROXY=12 /* runtime-internal */ };
```

Payload interpretation (`lv_abi.h:58-64`): `LV_FLOAT` stores the double bit-pattern in
`payload` (memcpy, never a cast); `LV_BOOL`/`LV_INT`/`LV_CHAR` store the value directly; tags
4/5/6/7/9/11 store heap pointers. Immediates (int, float, bool, char, void, none) marshal to
JS with **no heap access at all** — the fast path.

### 3.2 The heap header, and the two exceptions that will bite the marshaler

Every *counted* heap payload is preceded by a 16-byte header at `payload-16` (`lv_abi.h:66-78`):
`{ int64_t rc; int64_t meta; }`. **But two payload kinds carry no header** and are the classic
marshaler footgun:

- **Literal strings** — LLVM constant data, no `LvHeader`, exempt from ARC by a region-registry
  check (`lv_abi.h:85-93`). Confirmed in codegen: a literal is emitted as `StructType {i64 len,
  [N x i8] bytes}` with a trailing NUL and no header (`src/LlvmGen.cpp:490-501`).
- **Inline dense-array records** — a tag-5 object payload may point *into* a dense array buffer
  with no header (`lv_abi.h:61-63`). *"always isvalueclass()-gate before touching [payload-16]."*

The marshaler must gate on `isvalueclass()` / region-registry membership before ever
dereferencing `payload-16`. This is not wasm-specific — it is the existing rule — but a fresh
wasm marshaler is exactly where it gets re-forgotten.

### 3.3 Object shape and reflective slot walking

Object body (`lv_abi.h:95-107`): `payload → { int64_t classId; int64_t dyn; LvValue
slots[nslots]; }`, **slot `i` at byte offset `16 + 16*i`**. `dyn` heads a fallback linked list
of 32-byte `{next, keyPtr, valTag, valPay}` nodes for dynamic keys (in practice only closure
captures hit this).

Name→slot resolution is `slotNames[i]` in `LvClassInfo` (`lv_abi.h:227-246`) — `classId` finds
the class descriptor, `slotNames[i]` names the field at slot `i`, offset `16 + 16*i`. The
runtime helper `lvrt_fieldindex` does the lookup; slot-name comparison is pointer-first
(interned) with a strcmp content fallback (`lv_abi.h:256-260`). The interpreter-side twin that
matches this order is `columnarFieldSlot` (`Symbols.hpp:112-121`).

**This is the entire "marshal an object to JS" algorithm:** read `classId`, look up
`LvClassInfo`, for each `i in 0..nslots` emit `{ slotNames[i]: marshal(slots[i]) }`. One loop,
data-driven, no per-class code. Registered via `lvrt_register(const LvClassInfo* classes,
int64_t nclasses, LvDispatchFn dispatch)` (`lv_abi.h:262`) — the same table the marshaler reads.

### 3.4 Strings — the UTF-8↔UTF-16 boundary

Compiled representation (`lv_abi.h:85-93`): `LV_STR` payload → `{ int64_t len; char
bytes[len]; char nul; }`, bytes UTF-8. So a string is **ptr+len**. Marshaling to JS is
`TextDecoder.decode(new Uint8Array(memory.buffer, payload+8, len))`; from JS is
`TextEncoder.encode(str)` into a fresh runtime string. Two grounded caveats:

- **NUL deviation** (`lv_abi.h:91-93`): x64 descriptors carry no trailing NUL; the C runtime
  (`lv_runtime.c`) and LLVM literals do. The wasm target rides the LLVM/C-runtime lane, so it
  gets the NUL — but the marshaler must key off `len`, never NUL-termination.
- This is where the language's **open UTF-8/UTF-16 encoding question** (info.md §9/§10, what
  `subStr` counts) finally forces a ruling — the wasm target is the forcing function (proposal
  risk #2). Pin round-trips in a corpus.

### 3.5 Value structs vs reference objects; the three array layouts

The distinction is `LvClassInfo.isValue` / `Symbol::isValue` (`lv_abi.h:231`,
`Symbols.hpp:68`): value structs are copied, never refcounted (`copyValue`,
`RuntimeValue.hpp:117-124`; runtime `lvrt_copyval`). Arrays have **three on-heap shapes**,
discriminated by the top two bits of body word0 (`lv_abi.h:109-150`):

- **BOXED** (`lv_abi.h:111-116`): `{ int64_t len; LvValue elems[cap]; }`, both top bits clear.
  COW on `idxset` when `rc==1`. The universal path; every element is a full tagged `LvValue`.
- **DENSE** (`lv_abi.h:117-130`): `{ lenWithBit=cleanLen|(1<<63); int64_t recBytes; uint8_t
  records[...]; }`. Each record is an inline object body; reading an element yields a tag-5
  `LvValue` whose payload **points into the buffer** (the no-header case, §3.2). Value-struct
  elements only.
- **COLUMNAR** (struct-of-arrays, `lv_abi.h:131-150`, `techdesign-columnar-arrays.md`):
  `{ lenWithBits=cleanLen|(1<<63)|(1<<62); int64_t classId; uint8_t columns[...]; }`. Field
  `k` of element `i` is the raw payload word at `payload + 16 + k*8*cleanLen + 8*i`; tags
  synthesized on read from `lv_col_typecode`. Never hands out interior pointers.

**Marshaling relevance (the §2 zero-copy claim):** a DENSE or COLUMNAR `Array<Point>` of plain
scalars is *exactly* the layout a JS `TypedArray` view reads without a copy. Columnar
eligibility (`Symbols.hpp:98-108`): a value struct with ≥1 field, every field a plain scalar
int/float/bool/char, none weak/distinct. Typecodes deliberately equal the LV tags
(`Symbols.hpp:86-92`) so "tag synthesis on gather is a copy, not a mapping." **But** — this
dense-layout work is flagged "next major work" (proposal risk #7): the reference-generic
(BOXED) path works today without it; **ship boxed first, zero-copy when dense matures.**

Closure body (`lv_abi.h:157-166`): `{ int64_t fnIndex; int64_t captureHead; }` — reuses the
object dyn-list for captures. This is the representation the DOM-event closure trampoline rides
(§8.3). Map: `{ int64_t len; LvPair entries[len]; }` (`lv_abi.h:152-155`). Block (byte buffer):
`{ parentPtr, off, len, dataPtr }` (`lv_abi.h:168-177`), the natural backing for a marshaled
`ArrayBuffer`.

## 4. The wasm backend — LLVM → wasm32

The proposal's primary path is *LLVM → `wasm32`*, on the grounds that "the triple is already a
parameter." The tree confirms this is not just true — **it is already reachable with zero
compiler changes.**

### 4.1 `emitObject` already takes a triple and initializes all targets

```cpp
bool emitObject(const std::string& path, const std::string& triple = "",
                int optLevel = 2);              // src/LlvmGen.hpp:32-33
```

Inside (`src/LlvmGen.cpp:3309-3333`), target initialization is split: host emission registers
only the native target, but **any non-empty triple triggers the full registry**:

```cpp
if (tripleArg.empty()) {
    InitializeNativeTarget();
    InitializeNativeTargetAsmPrinter();
} else {
    InitializeAllTargetInfos();
    InitializeAllTargets();          // <-- WebAssembly registered here, transitively
    InitializeAllTargetMCs();
    InitializeAllAsmPrinters();
}
std::string triple = tripleArg.empty() ? sys::getDefaultTargetTriple()
                                       : Triple::normalize(tripleArg);
const Target* target = TargetRegistry::lookupTarget(triple, err);
if (!target) { sink_.error({}, "LLVM backend: " + err); return false; }
TargetMachine* tm = target->createTargetMachine(
    triple, "generic", "", TargetOptions(), Reloc::PIC_, std::nullopt, ...);
g.module->setDataLayout(tm->createDataLayout());
g.module->setTargetTriple(triple);
```

Because the code calls `InitializeAllTargets()` (not a hand-picked subset) and CMake links the
**whole LLVM 18 dylib** (`target_link_libraries(langfront PUBLIC LLVM)`, `CMakeLists.txt:49-61`),
every target the installed LLVM was built with is registered — and `llvm-config-18
--targets-built` includes `WebAssembly`. So `--target wasm32-unknown-unknown` reaches
`lookupTarget` successfully **today**, no CMake or codegen change required. There is literally
zero `wasm`-specific token anywhere in `src/`, `runtime/`, or `CMakeLists.txt` — the single
`InitializeAllTargets()` at `src/LlvmGen.cpp:3317` is the entire reason wasm is reachable.

### 4.2 CLI surface and the one codegen caveat

The backend is driven from `src/main.cpp` (arg parsing `:269-294`):

- `--emit-llvm` → textual IR (`gen.emitIr()`, `:578`)
- `--native-obj <out>` → object only (`gen.emitObject(outPath, targetTriple, optLevel)`, `:581`)
- `--build-native <out>` → object + link (`:597`)
- `--target <triple>` → the cross triple (default `""`, `:206`)
- `--opt-level <0|2>` (default 2, `:207`), `--runtime <path>` → archive override

For the spike, the minimal invocation is `--native-obj out.o --target wasm32-unknown-unknown`
— object emission only, sidestepping the not-yet-wasm-aware linker/runtime plumbing (§4.3).

**The one grounded codegen caveat:** the TLS-model pin (`src/LlvmGen.cpp:3343-3355`) is
format-dependent (ELF→InitialExec, COFF→non-TLS, else GeneralDynamic) and has **no wasm case**
— wasm would fall to `GeneralDynamicTLSModel`. Thread-locals are how the runtime keeps
per-thread heap/arena/loop state (§7.2); on a single-threaded first wasm build this is inert,
but it is the one place `emitObject` is silently wasm-naive and should be an explicit branch in
the real target.

### 4.3 What is *not* free — the runtime side

Codegen is free; the **runtime and link plumbing are not**:

- `runtime/build-triple.sh` branches only `*windows*|*mingw*` → `lv_plat_win32.c` vs POSIX
  (`:72-76`), and always adds the x86_64/aarch64 `.S` context-switch files. A `wasm32` triple
  currently falls through to the POSIX floor + native asm — **all wrong for wasm**. This is the
  §6 floor-retarget work: a new `lv_plat_wasm.c` and a wasm branch here.
- `src/main.cpp` link driving (`:600-673`) probes a native/gnu/clang cross linker, not
  `wasm-ld`. The real target needs a wasm link lane; the spike avoids it by stopping at `.o`
  and linking with `wasm-ld`/`wasmtime` externally.

### 4.4 The two fallback paths (in build order, per proposal §5)

- **emit-C++ → Emscripten** (bring-up lane only): `src/CGen.cpp` already emits freestanding
  C++; piping through Emscripten yields a browser build quickly and dissolves the "no libc++"
  worry (Emscripten supplies shims). Cost: an Emscripten toolchain dependency and larger
  output — it reintroduces exactly the dependency the LLVM-primary portable pivot shed
  (`techdesign-portable-backend.md:14-23`). **Never the shipping path** (proposal STOP
  condition; risk #4). Keep it labeled as a crutch.
- **Own `.wasm` emitter** (self-host analog, later): mirrors the pure-ELF backend's own
  machine-code + ELF writer — a `.wasm` module writer with no external toolchain. The
  philosophically-aligned endgame, **not a first step**, and explicitly *not* routed through
  the frozen `X64Gen.cpp`/`X64.hpp` — a new emitter alongside the LLVM path.

## 5. The de-risking spike — a concrete runbook

The proposal's §9 spike ("retarget LLVM→`wasm32`, run pure computation, differential-match the
oracle") is the one thing to do now. It is smaller than the proposal estimated (1–2 days),
because **`liblvrt.a` has since landed** (§12): the runtime the pure-compute subset references
is real, not a stub.

### 5.1 Why it is now nearly mechanical

The proposal assumed the spike would link "the existing `llvm_stub_runtime.c`." **That file no
longer exists** — `CMakeLists.txt:826-827` states the stub is retired and the LLVM backend now
links the real runtime. `liblvrt.a` is a real target (`CMakeLists.txt:1198-1205`, aggregating
`lv_runtime.c` + `lv_plat_posix.c` + loop/thread/entry + task substrate), and built archives
exist on disk (host `build/liblvrt.a`, cross `runtime/aarch64-linux-gnu/liblvrt.a`, etc.). The
allocator, ARC, string, and object runtime the pure-compute subset needs are all in
`lv_runtime.c`, target-independent.

### 5.2 Runbook

1. **Emit the object.** `leviathan --native-obj prog.o --target wasm32-unknown-unknown prog.lev`
   — works today, per §4.1. (Confirm: `wasm32-wasi` is the alternative triple if a minimal WASI
   shim is easier than a hand loader.)
2. **Provide a minimal runtime.** Compile the pure-compute subset of `lv_runtime.c` to wasm
   (`clang --target=wasm32 -c lv_runtime.c` for the allocator/ARC/value ops), OR hand-write a
   tiny linear-memory allocator stub for just the `lvrt_*` the subset references. The §15
   allocator (ARC + per-frame arena + free-list) runs over a heap region — point it at a
   `WebAssembly.Memory` and `mmap`-growth becomes `memory.grow` (§6.2). **No host call is
   needed to allocate** — the runtime owns its allocator.
3. **Link and run.** `wasm-ld prog.o lvrt_wasm.o -o prog.wasm --no-entry --export-all`, then
   run in `wasmtime` or a trivial browser loader.
4. **Differential-match.** Run a pure-computation corpus subset — objects, generics, `struct`s,
   arrays, closures, arithmetic; **no I/O, no async, no DOM** — through the wasm module and the
   IR interpreter oracle (`src/IrInterp.cpp`); assert identical observable output.

### 5.3 Success / failure semantics

- **Success** = the uniform tagged-value representation compiles and runs on wasm and needs no
  monomorphization — empirically closing the §2 type-system question for near-zero cost, without
  touching the floor, async, or bindgen.
- **Failure** = the LLVM backend cannot emit working `wasm32` without runtime-ABI work that is
  really the portable-backend owner's — a cheap early signal. **STOP condition** (proposal §12):
  coordinate with the portable-backend owner, do **not** fork the runtime.

The scratch space for this belongs in a throwaway branch; the spike commits to nothing.

## 6. The runtime-floor retarget — `lv_plat_wasm.c`

This is the "real runtime work," and it is remarkably contained because the floor is *already*
a single, enforced seam.

### 6.1 The floor is one file's worth of symbols, and nothing bypasses it

`runtime/lv_plat.h:1-4` states the invariant verbatim:

> Track B — the platform floor. The ONLY place the runtime touches an OS (doc-2 §0.1 item 4).
> lv_runtime.c / lv_loop.c must never call an OS primitive directly — everything routes through
> here so a new target (macOS, Windows, aarch64) is a new lv_plat_*.c file, nothing else.

It declares ~40 functions (`lv_plat.h:22-142`): `lv_plat_map/unmap` (anonymous RW mapping),
`lv_plat_write/read`, `lv_plat_now_ns/now_realtime_ms`, `lv_plat_exit`, `lv_plat_open/close`,
`lv_plat_random` (CSPRNG bytes), the socket family (`tcp_connect/listen/accept/send/recv`,
nonblocking variants, `connect_result`), `lv_plat_poll` over the opaque `LvPollFd`, terminal
raw-mode, signals-as-fd, and filesystem metadata. **A wasm target is `lv_plat_wasm.c`
implementing these same symbols against host imports — nothing else in the runtime changes.**
POSIX is 499 lines, Win32 is 459; the wasm floor is the same order of magnitude, minus the
capabilities a browser lacks (§6.3).

The seam is real, not aspirational. Every `lvrt_sys*` native bottoms out in `lv_plat_*`, e.g.:

```c
void lvrt_syswrite(LvValue* out, const LvValue* fd, const LvValue* s) {   // lv_runtime.c:2521
    ... lv_plat_write((int)fd->payload, P8(s->payload)+8, n); ...
}
void lvrt_sysnow(LvValue* out) { out->tag = LV_INT;                       // lv_runtime.c:2571
    out->payload = lv_plat_now_realtime_ms(); }
```

Even `print` and the panic writer go through `lv_plat_write` (`lv_runtime.c:2511,2516,203`),
and the process entry `lv_entry.c` touches no OS primitive directly (`:17-19`). Grepping
`lv_plat_` across `lv_runtime.c` shows the *only* OS surface is `lv_plat_*`. This is the §8
"standing discipline" already holding: portable code never calls a floor native directly.

### 6.2 The per-op wasm substitutions

| floor function | POSIX today (`lv_plat_posix.c`) | wasm-browser (imported from JS `env`) |
|---|---|---|
| `lv_plat_write(fd,buf,n)` | `write(2)` (`:38-40`) | `console.log` / a ring buffer to a `<pre>`; fd 1/2 → stdout/stderr |
| `lv_plat_map/unmap` | `mmap`/`munmap` | grow `WebAssembly.Memory` via `memory.grow`; the §15 allocator owns the region (§6.4) |
| `lv_plat_now_realtime_ms` | `clock_gettime` | `Date.now()` (imported) |
| `lv_plat_now_ns` | monotonic clock | `performance.now()*1e6` (imported) |
| `lv_plat_random(buf,n)` | glibc `getrandom` (`:450-470`) | `crypto.getRandomValues` (WebCrypto) |
| `lv_plat_poll(fds,n,timeout)` | `poll(2)` (`:472-499`) | event-loop hooks — resolve on host promise settle / `setTimeout` (§7) |
| `lv_plat_exit(code)` | `_exit` | set an exit flag; there is no process to exit — return to the host |
| `lv_plat_open/close/stat/mkdir` | file syscalls | **absent** — compile-time diagnostic (§6.3) |
| `lv_plat_tcp_*`, `accept`, raw `send/recv` | socket syscalls | **absent as raw sockets** — reshaped to `fetch`/`WebSocket` stream endpoints (§6.3, §8) |
| `lv_plat_term_*`, `signal_*` | tty/signalfd | **absent** — no tty, no signals in a browser |

The `LvPollFd` struct (`lv_plat.h:140-142`, `{ int fd; int16_t events; int16_t revents; }`
with `LV_POLLIN/OUT/NVAL` bits) is deliberately opaque precisely so a non-POSIX floor can
re-back it — Win32 already maps it onto `WSAPOLLFD`. The wasm floor maps the same three bits
onto its readiness model (host-promise settlement), no caller sees a raw `POLLIN`.

### 6.3 The capability subset falls out — it is not hand-carved

Two layers, not to be conflated (proposal §3.1):

- **Present on both targets** (console, time, random, the event loop, networking): the *same
  stream surface wired to a different floor* — a silent runtime swap, invisible to the program.
- **Absent on a target** (files, processes, raw sockets, ttys, signals in a browser): caught at
  **compile time** by the capability gate, with a diagnostic.

Crucially, most of the OS-privileged floor is *inherently* meaningless in a browser sandbox —
processes, directories, ttys, raw sockets — so the subset is natural, not a carved-out dialect.
You lose *raw* sockets, not *networking*: HTTP reshapes onto `fetch`, sockets onto `WebSocket`,
both as §13 stream endpoints (§8.5).

### 6.4 The allocator ports nearly free

The §15 memory story — ARC (retain/release), per-frame arena (reset at return), free-list,
COW-on-refcount — runs over a heap region and is entirely target-independent. Point it at a
`WebAssembly.Memory`; `mmap`-growth becomes `memory.grow`. Because Leviathan owns its allocator,
**no host call is needed to allocate**. This is the cheapest part of the whole target.

### 6.5 The clean-diagnostic pattern — grounded, and its one nuance

The proposal cites ELF-DNS as the model: the pure ELF backend has no libc resolver, so
`sysResolve` "emits *'resolve: not on the pure backend yet'* rather than faking it." The
grounded reality (worth pinning so the wasm design copies the *mechanism*, not the prose):

- The exact string *"resolve: not on the pure backend yet"* is the **design's** phrasing
  (`techdesign-08-system-natives.md:117-121`; `RuntimeNatives.cpp:1846-1849`) and does **not**
  appear verbatim in code.
- The mechanism is: an unsupported capability hits `X64Gen::fail(...)` at the native-dispatch
  fallthrough (`src/X64Gen.cpp:3859-3860`), or a bespoke `fail()` — e.g. threads
  (`X64Gen.cpp:3837`: *"threads: unsupported on the ELF backend (use LLVM)"*), transcendental
  math (`:3640`). Compile-time diagnostic, never a silent drop.
- Pinned as permanent policy for the frozen ELF lane (`deferal-http-and-streams-maturity.md:36`).

The wasm backend implements the same pattern: referencing `File` in browser-targeted code
becomes a compile-time diagnostic at the wasm backend's native dispatch, exactly like ELF-DNS.

### 6.6 The seam reaches bare metal too (why §8's discipline pays double)

The floor retarget is a *general* seam, not a wasm trick (proposal §3.5). The `syscall`-based
floor is merely the *Linux-userspace implementation* of Layer 0. It retargets in two
directions: **sideways → the browser** (syscall → imported JS functions, this dossier), and
**downward → bare metal** (syscall → *nothing beneath*: MMIO serial/framebuffer, allocator
bootstrapping from physical page frames, no kernel to call because you are writing it). Same
shape — a backend column + a floor retarget, language and IR untouched. Every capability kept
behind `lv_plat.h` is one reimplementable on either target; every place library code reaches
past it to assume "a Linux is here" is a place both wasm and OS work would break. That is why
the §8 discipline is the cheap-now, high-value move.

## 7. Async and threads on wasm — the substrate the proposal called the pacing item, now landed

This is the section the ground moved under. The proposal (2026-07-08) named async suspension
as *"the single largest line item, and the reason the target waits for the §19 #5 async
decision."* **That decision has been made and the machinery has landed** (audited 2026-07-16).

### 7.1 True stackful suspension is now the default on all three engines

The suspension track landed 2026-07-11/12 (`designs/suspension/handoff-atlantis-kernel.md:1`,
filed at the M5 flip 2026-07-12; task-substrate doc moved to `designs/complete/`). `info.md`
§14 states it plainly (`info.md:1642-1648`):

> `await` … is a **suspension point** … Since LA-30 that machinery is **stackful tasks** on all
> three active engines: the awaiting task genuinely suspends (its own stack parks; a per-thread
> scheduler runs whatever else is runnable) and resumes in completion order…

The old pump-until-ready model survives only behind `LANG_PUMP=1`, scheduled for deletion at
M6. `lvrt_await` (`lv_runtime.c:2985`) now *parks* the task rather than pumping:

```c
void lvrt_await(LvValue* dst, const LvValue* p) {                  // lv_runtime.c:2985
    if (p->tag != LV_OBJ) { *dst = *p; return; }   /* await non-promise = identity */
    ...
    if (!(lvrt_promise_state(obj) & 1)) {          /* not ready */
        int r = lv_tasks_enabled()
              ? lv_task_park_on(obj)                /* park: stackful */
              : lv_pump_until_ready(p);             /* LANG_PUMP=1 only, deleted at M6 */
    ...
```

### 7.2 Leviathan already does the exact thing JSPI does — this is the whole point

The proposal's key claim — *"JSPI is browser-native stackful suspension, the same model §14
already implies"* — is not analogy; it is the same operation. Leviathan already parks and
resumes real call stacks via hand-written per-arch context switches. `runtime/lv_ctx_x86_64.S`
and `lv_ctx_aarch64.S` both export `lv_ctx_switch(void** save_sp, void* load_sp)`, saving
callee-saved GPRs + FP env, swapping the stack pointer, and returning into the target context:

```asm
lv_ctx_switch:                     # lv_ctx_x86_64.S:35 — save r15..rbp, fp env; swap rsp; ret
    pushq %rbp ; pushq %rbx ; pushq %r12 ; pushq %r13 ; pushq %r14 ; pushq %r15
    subq  $8, %rsp ; stmxcsr 4(%rsp) ; fnstcw (%rsp)
    movq  %rsp, (%rdi)             /* *save_sp = sp  */
    movq  %rsi, %rsp              /* sp = load_sp   */
    ... popq %r15 ... popq %rbp ; ret     /* returns into the target context */
```

Driven from `lv_task.c:258-261` (`ctx_to_task`/`ctx_to_sched`); each task is an `mmap`'d stack
with a `PROT_NONE` guard page (`task_new`, `:307-315`), hand-seeded to mirror the `.S` frame so
the first switch-in `ret`s into a trampoline (`task_seed`). The per-thread scheduler
(`lv_sched_run`, `:932-962`) pops a FIFO run queue in completion order and settle-scans parked
tasks (`poll_parked`, `:844-865`).

**On wasm, that ~12-instruction register/stack swap is exactly what the engine's JSPI does for
you.** When a suspending import returns a JS promise, JSPI parks the whole wasm stack and
resumes it on resolve — the same "park this stack, run the scheduler, resume later" operation,
one rung lower (engine-provided instead of assembly-provided). One *model* (stackful), two
realizations, because stack-switching is inherently target-specific. On the native backend the
same model is these `.S` fibers; on wasm-browser it is JSPI. This is why the pump→tasks swap
needed **no surface change** and why JSPI will need none either.

### 7.3 The no-coloring commitment is what makes JSPI (not CPS) the right fit

`info.md:1513-1515` (§13): *"There is **no function coloring**: any function may `await`, and a
function is 'async' simply by returning a `Promise<T>` (no `async` keyword)."* This is a
**locked-in** surface (`info.md:1634-1640`): `Promise<T>` is a plain object library, only
`await` is privileged.

The mechanism choice follows directly:

- **Stackful (JSPI, and Leviathan's fibers)** suspends the *whole stack*, so *any* function can
  await with no per-function transform — matches no-coloring exactly.
- **Stackless CPS** (`llvm.coro`, C#/Rust/JS async) splits each async function into a state
  machine and therefore *needs* coloring to know which to split — "coloring at the IR level,"
  explicitly **foreclosed, not deferred** (`techdesign-00-overview.md:181-186`).

So a stackless CPS transform on wasm is **not recommended** — it would reintroduce the exact
contagion §14 rejected. The open decision the proposal deferred to (§19 #5, async coloring) has
resolved in favor of the uncolored form, and the handoff doc names the survival of uncolored
`await` across the pump→tasks swap as *"the strongest evidence the claim will hold"*
(`handoff-atlantis-kernel.md:24-26`). **The async model is now built once, correctly, ahead of
the wasm target** — the reverse of the proposal's worry.

### 7.4 Bring-up fallback: Asyncify

JSPI's only cost is a *platform-support* dependency (engine feature, shipping but not universal).
The works-today fallback is **Asyncify** — a Binaryen/Emscripten build-time whole-program
rewrite that delivers stackful suspension with no engine support, at a real code-size and speed
cost (instrumentation ships in the module) plus an Emscripten dependency. Same category as
emit-C++→Emscripten (§4.4): a crutch while JSPI support settles, never the shipping path.

### 7.5 Threads → Web Workers, already a message-passing runtime

The threads surface — `spawn` (low tier, `info.md:1612-1621`), `Worker<T> : Promise<T>` (high
tier), isolation-by-default (`info.md:1623-1628`) — is unchanged; only the mechanism swaps. And
the landed model is *already* message-passing, which maps onto Workers almost 1:1:

- Every value crossing a thread boundary crosses by **deep copy** — a flatten/rebuild engine
  (`lv_thread_flatten`/`rebuild`, `lv_thread.h:23-25`) with a seen-map for shared
  substructure/cycles. This **is** structured-clone.
- **No shared heap** — "a counted value lives on exactly one unit," non-atomic refcount fast
  path preserved (`info.md:1602-1603`). Each worker gets its own TLS heap/arena/event loop, real
  `pthread`s, an `eventfd` join (`info.md:1596-1599`). The `eventfd` join **is** `postMessage`
  completion.
- Tasks are **thread-pinned forever**, no migration/work-stealing (`lv_task.h:1`); "a
  continuation is only ever run by the thread that owns it" — enforced by Workers *by
  construction*.
- Handles may not cross; `Channel` is the one portal (`info.md:1603-1604`). Windows/ELF already
  reject `spawn`/`Channel` at compile time (`info.md:1606-1607`), so a wasm leg slots in as
  another gated platform.

The mapping: OS threads → **Web Workers** (each its own wasm instance); the gated shared-mutable
path → **SharedArrayBuffer + Atomics**. The one casualty is low-tier raw clone-style `fork` with
a shared address space — but §14 already gates that as Unix-specific, so a browser simply omits
that gate (a subset). The high tier — what people actually use — maps cleanly.

## 8. The JS/DOM bridge — the `wasm-bindgen` equivalent

Mostly a **library + rules** deliverable, not a compiler subsystem. Four pieces.

### 8.1 A handle table for opaque JS values

DOM nodes/events are opaque JS values held either as `externref` (reference-types, widely
shipped) or as integer indices into a JS-side `heap[]`. A Leviathan wrapper:

```
class DomNode {
    handle h;      // an i32 index into the JS-side heap[], or an externref slot
}
```

DOM APIs (`getElementById`, `setAttribute`, `addEventListener`) are imported functions over
handles. The handle is opaque to Leviathan — it never dereferences it; only the JS glue does.

### 8.2 One reflective marshaler, off tag + shape (written once)

This is the §2/§3 payoff made concrete. The marshaler is a single table-driven routine:

```
marshal(LvValue* v) → JS value:
  switch (v->tag):                       // lv_abi.h:38-53
    LV_INT / LV_BOOL / LV_CHAR  → v->payload directly            (immediate, no heap)
    LV_FLOAT                    → reinterpret payload bits as f64 (lv_abi.h:58-64)
    LV_NONE / LV_VOID           → null / undefined
    LV_STR   → TextDecoder.decode(memory[payload+8 .. +8+len])   (UTF-8→UTF-16, §3.4)
    LV_OBJ   → classId = mem[payload]; ci = LvClassInfo[classId];
               { ci->slotNames[i]: marshal(slot i @ payload+16+16*i)  for i in 0..nslots }
    LV_ARR   → boxed: [ marshal(elems[i]) ]; dense/columnar scalars → TypedArray view (§8.4)
    LV_CLO   → register in the closure table, hand JS the index    (§8.3)
    LV_MAP   → { marshal(k): marshal(v) } over entries
```

Because the representation is **uniform**, this is written *once*, not regenerated per
instantiation the way `wasm-bindgen` must. `Array<string>` and `Array<DomNode>` share one
runtime type and one marshaling path — the type argument was a compile-time slot, checked and
gone; the container holds tagged values (`lv_abi.h` boxed array). Reference generics are
erased/uniform (like every VM); value generics are reified (§8.4). Remember the two
no-header exceptions (§3.2) — gate `isvalueclass()` before touching `payload-16`.

### 8.3 A closure trampoline for events

Rides the existing closure representation (`lv_abi.h:157-166`, `{ fnIndex, captureHead }`).
Export a single trampoline; register Leviathan closures in a table; hand JS the index; JS calls
the trampoline per DOM event:

```
export wasm_invoke_closure(closureHandle: i32, eventHandle: i32):
    clo   = closureTable[closureHandle]           // LV_CLO value, captures intact
    event = new DomEvent{ h = eventHandle }        // handle-wrapped opaque JS event
    call clo with (event)                          // ordinary CallValue over the closure
```

`addEventListener("click", cb)` → JS stores `cb`'s closure index and installs a real DOM
listener that calls `wasm_invoke_closure(idx, evt)`. The closure's captures (the object-graph
it closed over) are already in linear memory via `captureHead` — nothing extra to marshal.

### 8.4 The zero-copy typed path (ship boxed first)

A value-struct `Array<Point>` in DENSE or COLUMNAR layout (§3.5) of plain scalars is exactly
the byte layout a JS `TypedArray` reads without a copy — the one place concrete layout genuinely
matters, covered *on demand* without making everything reified. **But** dense/columnar beyond
plain structs is "next major work" (proposal risk #7): the BOXED path works today. **Ship the
boxed marshaler first; add the zero-copy TypedArray view when the dense-layout work matures.**
`IrModule::columnar` (`src/Ir.hpp:130`) is the gate; a first wasm backend can leave it off.

### 8.5 DOM events as streams

§13 already makes streams the system boundary — sockets/timers are stream endpoints,
`EventEmitter` is Layer-2 fan-out. A DOM event source → a stream of events is idiomatic, not
bolted on: the browser event loop replaces the native one under the same surface. Networking
reshapes the same way — HTTP over `fetch`, sockets over `WebSocket`, both as stream endpoints.

### 8.6 Generating the bridge with the rules engine — and it is *more* ready than the proposal assumed

The proposal flagged that the ergonomic bindgen "wants metaprog Phase 3+ splices/reflection,
which are unbuilt (§16.5)." **Phases 1–4 + F4 have since landed** (`info.md:1818-1821`:
*"Phases 1–4 and F4 procedural macros are implemented."*). So the bindgen's dependencies —
`meta.*` reflection, `$for` list splices, the `namespace N` anchor, def-site qualification —
are **all available today** (§12).

The bindgen is a rules artifact. The model already in the tree is `@Test` auto-discovery
(`techdesign-unit-test-library.md:174,187-196`), which does *exactly* what a bindgen does —
emit one generated declaration per annotated method into a namespace registry:

```
attribute Test { }     // pure marker
rule discoverTests {
    match on class C
    where C.methods.any((x) => x.hasAttr("Test")) && !C.hasBase("IDisposable")
    inject `bool $C = harpoon::__seed("$C",
        [ $for m in C.methods.where((x) => x.hasAttr("Test") && x.returnType == "void" && x.arity() == 0)
            : harpoon::__entry("$m", () => { $C t = $C(); t.$m(); }) ], ...);`
    at namespace HarpoonReg
}
```

A DOM bindgen maps 1:1. Match `@extern("dom") class C`, `$for m in C.methods` emitting a
marshaling stub per method into `namespace DomBindings`:

```
namespace Dom {
    attribute extern { string kind; }   // @extern("dom")
    rule bindExtern {
        match @extern(e) on class C
        inject `$for m in C.methods : Dom::__import($e.kind, "$m", ($_params) => this.$m($_args))`
        at namespace DomBindings
    }
}
```

The grounded primitives this uses (all landed):

- **Attribute match binds the value as an Object** (`Rules.cpp:1478-1508`) — `@extern(e)` binds
  `e`; `$e.kind` reifies the string field (`Rules.cpp:2035-2060`). **Important nuance**
  (`Rules.cpp:1421-1424`, `techdesign-metaprog-phase3.md:253-254`): `meta.*` reflects attribute
  **names only**, not their arguments — so the bindgen must *match* `@extern(e)` to read its
  payload, never reflect it. This shapes the rule structure.
- **`$for` over `C.methods`** (`meta::Class.methods`, `Rules.cpp:1390-1444`) emits one stub per
  method; `$m`/`$C` name splices (`Rules.cpp:2038-2048`), `t.$m()` selector dispatch
  (`Rules.cpp:2094-2125`), `$_params`/`$_args` forwarding (`Rules.hpp:349-355`).
- **`at namespace N`** — the generated-decl channel (`Rules.cpp:1818-1829`): a cloned
  `namespace` Stmt accumulates in `namespaceInjections_` (`Rules.hpp:374`) and is flushed into
  `program.items` at end of `run()` (`Rules.cpp:113-115`), then goes through pass-2
  resolve/check like hand-written code — **generated declarations are cost-identical to
  hand-written ones** (`info.md:1853-1856`).
- **Def-site qualification** (`techdesign-metaprog-phase3.md:530-567`) so injected references to
  the bindgen library's own runtime helpers (`Dom::__import`) survive splicing into foreign user
  files.
- Layer D `rewrites`/`$body` (Phase 4) is available if a stub must *wrap* a method body
  (marshal-in / call / marshal-out) rather than only append.

For a proof-of-concept, hand-written import stubs are possible earlier and are throwaway
(proposal risk #6); with Phases 1–4 landed, the ergonomic generated path is available for the
real target without waiting.

## 10. String encoding, memory, and cycles — the standing hazards

- **String encoding** (§3.4, proposal risk #2): Leviathan strings are UTF-8 handles; JS is
  UTF-16. Transcode at the marshaler (`TextEncoder`/`TextDecoder`). The wasm target is the
  forcing function that finally rules on the open §9/§10 encoding question (what `subStr`
  counts). Pin round-trips in a corpus.
- **Reference cycles** (proposal risk #3, info.md §15/§19 #10): a DOM node ↔ Leviathan closure
  handler is exactly the Timer→handler cycle shape the escaping-tier ARC does not collect. This
  is **pre-existing, not wasm-specific**. Follow the churn-corpus discipline (bind before
  holding); the `weak`-ref-vs-cycle-collector decision (§19 #10) becomes *more* pressing once
  DOM handler graphs exist. Do not let the bridge invent its own leak — note the `LV_WEAK_PROXY`
  tag (`lv_abi.h:38-53`) and `isWeak` slots (`Symbols.hpp:52`) already exist as the weak-field
  substrate.

## 11. What each target loses, keeps, gains (grounded, per-target — not the language's loss)

Strictly per-target. **Native Leviathan loses nothing.** The "Lost" column is the *wasm-browser
target's* smaller floor only — same way Rust's `wasm32` has no `std::fs` while Rust-native does.

| | wasm-browser target |
|---|---|
| **Lost** (compile-time gated, §6.3/§6.5) | filesystem (`File`, dirs), process spawning (`sysSpawn`/`Process`), raw TCP/UDP + `sysResolve` DNS, argv/env, tty/`isatty`, synchronous blocking reads, raw OS threads / shared-address `fork` |

| **Gained** (via the §8 bridge) | the DOM, `fetch`, `WebSocket`, WebCrypto, Canvas/WebGL, storage (localStorage/IndexedDB/OPFS) |
